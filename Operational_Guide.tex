\section{Purpose and Scope}
% =================================================================

This guide contains all information required to reproduce the project
\textit{Air Quality Assessment in Torino} on a clean machine.
It covers every operational phase in order:

\begin{enumerate}
  \item Prerequisites and local environment setup
  \item Google Cloud Platform (GCP) configuration
  \item Credential management
  \item Data acquisition via dlt pipelines
  \item Data transformation and quality checks via dbt
  \item Analytical SQL queries
  \item Streamlit dashboard execution
  \item Troubleshooting reference
\end{enumerate}

\begin{tcolorbox}[infobox]
\textbf{Reproducibility guarantee.}
All commands in this guide have been tested on Windows 11 with PowerShell 5.1
and Python~3.13.9 inside a local \texttt{.venv} virtual environment.
The GCP project identifier is \texttt{dm-torino-airquality}.
\end{tcolorbox}

\subsection{Python virtual environment}

From the project root, create and activate the virtual environment:

\begin{lstlisting}[style=shell]
# Create
python -m venv .venv

# Activate (PowerShell)
.\.venv\Scripts\Activate.ps1

# If execution policy blocks the script:
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
\end{lstlisting}

Install all required packages:

\begin{lstlisting}[style=shell]
pip install dlt[bigquery] dbt-bigquery python-dotenv requests
pip install streamlit pandas plotly google-cloud-bigquery db-dtypes pyarrow
\end{lstlisting}

\begin{tcolorbox}[warnbox,title=Note on execution policy]
On Windows, PowerShell may block \texttt{.ps1} script execution.
Run \texttt{Set-ExecutionPolicy -Scope CurrentUser RemoteSigned} once before activating the virtual environment.
\end{tcolorbox}

\subsection{Service account}

\begin{lstlisting}[style=shell]
# Create service account
gcloud iam service-accounts create dlt-loader `
  --description="Service account for dlt data ingestion" `
  --display-name="dlt loader"

# Assign BigQuery roles
$SA = "dlt-loader@dm-torino-airquality.iam.gserviceaccount.com"
gcloud projects add-iam-policy-binding dm-torino-airquality `
  --member="serviceAccount:$SA" --role="roles/bigquery.dataEditor"
gcloud projects add-iam-policy-binding dm-torino-airquality `
  --member="serviceAccount:$SA" --role="roles/bigquery.jobUser"

# Generate JSON key
New-Item -ItemType Directory -Force -Path "$HOME\gcp-keys"
gcloud iam service-accounts keys create `
  "$HOME\gcp-keys\dlt-loader-key.json" `
  --iam-account="$SA"
\end{lstlisting}

\section{Data Acquisition — dlt Pipelines}

All three pipelines must be run from the project root with the virtual environment

Run the pipelines in the order shown below.
The reverse-geocoding pipeline (Step~3) reads from \texttt{staging\_marts.dim\_station},

  AND ReportingYear BETWEEN 2022 AND 2024

\section{Analytical SQL Queries}
% =================================================================

All queries in the \texttt{queries/} directory target the
\texttt{raw.air\_quality\_statistics\_torino} table.
Run them from the project root using the \texttt{bq} CLI:

\begin{lstlisting}[style=shell]
# Recommended: pipe the file to bq (avoids multiline quoting issues)
Get-Content -Raw .\queries\annual_means_by_station.sql |
  bq query --use_legacy_sql=false
\end{lstlisting}

\begin{table}[h!]
\centering
\caption{Available SQL query files}
\small
\begin{tabularx}{\textwidth}{@{}lX@{}}
\toprule
\textbf{File} & \textbf{Description} \\
\midrule
\texttt{annual\_means\_by\_station.sql}             & Annual PM10 and NO2 mean by station, pollutant, and year \\
\texttt{annual\_means\_by\_station\_type.sql}       & Annual PM10 and NO2 mean aggregated by station type \\
\texttt{air\_quality\_station\_year\_summary.sql}   & Full KPI summary by station and year \\
\texttt{air\_quality\_station\_type\_year\_summary.sql} & Full KPI summary by station type and year \\
\texttt{pm10\_exceedances\_by\_station.sql}         & PM10 exceedance days by station and year \\
\texttt{pm10\_exceedances\_by\_station\_type.sql}   & PM10 exceedance days aggregated by station type \\
\texttt{check\_air\_quality\_count.sql}             & Row count verification query \\
\bottomrule
\end{tabularx}
\end{table}

\textbf{Example — annual means by station:}

\begin{lstlisting}[style=sql]
SELECT
  station_code, station_type, station_area,
  pollutant, reporting_year,
  pollution_level AS annual_mean_ugm3,
  data_coverage
FROM raw.air_quality_statistics_torino
WHERE aggregation_process_id = 'P1Y'
  AND unit = 'ug/m3'
  AND pollutant IN ('PM10', 'NO2')
ORDER BY station_code, pollutant, reporting_year;
\end{lstlisting}

\section{Purpose}

This document explains how to reproduce the project deliverables required by the Data Management course: data acquisition, storage, transformation, quality checks, analytical queries, and dashboard execution.

\section{Repository Structure}

Main folders:

\begin{itemize}
  \item \texttt{dlt\_pipelines/}: Python ingestion pipelines
  \item \texttt{dbt\_project/}: dbt models and tests
  \item \texttt{queries/}: analytical SQL queries
  \item \texttt{dashboard\_streamlit\_v2/}: Streamlit dashboard
\end{itemize}

\section{Prerequisites}

Required software:

\begin{itemize}
  \item Python 3.13 or compatible version available in the local virtual environment
  \item Google Cloud project with BigQuery enabled
  \item dbt Core with the BigQuery adapter
  \item a valid service account key for BigQuery access
\end{itemize}

\section{Credentials and Environment Variables}

The project requires Google Cloud credentials.

Expected variables:

\begin{itemize}
  \item \texttt{GOOGLE\_APPLICATION\_CREDENTIALS}
  \item \texttt{GCP\_PROJECT\_ID}
  \item optionally \texttt{BQ\_MARTS\_DATASET} if a dataset other than \texttt{staging\_marts} is used
\end{itemize}

Recommended PowerShell setup:

\begin{lstlisting}[language={}]
$env:GOOGLE_APPLICATION_CREDENTIALS = "C:\Users\Pc\gcp-keys\dlt-loader-key.json"
$env:GCP_PROJECT_ID = "dm-torino-airquality"
\end{lstlisting}

The dashboard also supports a fallback credential path at:

\begin{lstlisting}[language={}]
%USERPROFILE%\gcp-keys\dlt-loader-key.json
\end{lstlisting}

\section{Python Environment}

From the project root, use the existing virtual environment:

\begin{lstlisting}[language={}]
\.venv\Scripts\python.exe --version
\end{lstlisting}

If needed, install project dependencies for the dashboard layer:

\begin{lstlisting}[language={}]
\.venv\Scripts\pip.exe install -r dashboard_streamlit_v2\dashboard_streamlit_v2\requirements.txt
\end{lstlisting}

\section{Data Acquisition}

Run the three dlt pipelines from the project root.

\subsection{Open-Meteo}

\begin{lstlisting}[language={}]
\.venv\Scripts\python.exe dlt_pipelines\open_meteo.py
\end{lstlisting}

\subsection{EEA DiscoData}

\begin{lstlisting}[language={}]
\.venv\Scripts\python.exe dlt_pipelines\eea_discodata.py
\end{lstlisting}

\subsection{Reverse Geocoding}

\begin{lstlisting}[language={}]
\.venv\Scripts\python.exe dlt_pipelines\station_reverse_geocoding.py
\end{lstlisting}

Expected raw outputs:

\begin{itemize}
  \item \texttt{raw.air\_quality\_statistics\_torino}
  \item \texttt{raw.weather\_torino\_hourly}
  \item \texttt{raw.station\_reverse\_geocoding}
\end{itemize}

\section{Transformation Layer with dbt}

Move into the dbt project and run transformations and tests.

\begin{lstlisting}[language={}]
Set-Location dbt_project
..\.venv\Scripts\dbt.exe run
..\.venv\Scripts\dbt.exe test
\end{lstlisting}

Expected result:

\begin{itemize}
  \item staging and mart models materialized successfully
  \item 30 dbt tests passed
\end{itemize}

Main analytical outputs:

\begin{itemize}
  \item \texttt{staging\_marts.fact\_air\_quality\_annual}
  \item \texttt{staging\_marts.mart\_station\_type\_year\_summary}
  \item \texttt{staging\_marts.mart\_air\_quality\_weather\_yearly}
  \item \texttt{staging\_marts.dim\_station\_enriched}
  \item \texttt{staging\_marts.dq\_reverse\_geocoding\_quality}
\end{itemize}

\section{Analytical Queries}

The project includes several SQL files in the \texttt{queries/} folder. For example, from PowerShell:

\begin{lstlisting}[language={}]
Get-Content -Raw .\queries\annual_means_by_station.sql | bq query --use_legacy_sql=false
\end{lstlisting}

Other available queries include:

\begin{itemize}
  \item \texttt{annual\_means\_by\_station\_type.sql}
  \item \texttt{air\_quality\_station\_year\_summary.sql}
  \item \texttt{pm10\_exceedances\_by\_station.sql}
  \item \texttt{pm10\_exceedances\_by\_station\_type.sql}
\end{itemize}

\section{Dashboard Execution}

The outer dashboard folder contains a launcher so the dashboard can be started with a simple command.

From the project root:

\begin{lstlisting}[language={}]
Set-Location dashboard_streamlit_v2
..\.venv\Scripts\streamlit.exe run app.py
\end{lstlisting}

Alternative full-path command:

\begin{lstlisting}[language={}]
\.venv\Scripts\streamlit.exe run dashboard_streamlit_v2\dashboard_streamlit_v2\app.py
\end{lstlisting}

Expected local URL:

\begin{lstlisting}[language={}]
http://localhost:8501
\end{lstlisting}

\section{Troubleshooting}

\subsection{Missing Google Credentials}

Symptom: \texttt{DefaultCredentialsError} when opening the dashboard or running pipelines.

Resolution:

\begin{itemize}
  \item verify \texttt{GOOGLE\_APPLICATION\_CREDENTIALS}
  \item verify that the service account key file exists
  \item verify \texttt{GCP\_PROJECT\_ID}
\end{itemize}

\subsection{Wrong Dashboard Path}

Symptom: \texttt{streamlit run app.py} fails from the wrong folder.

Resolution:

\begin{itemize}
  \item move to the outer \texttt{dashboard\_streamlit\_v2} folder and run the launcher
\end{itemize}

\subsection{dbt or BigQuery Errors}

Recommended checks:

\begin{itemize}
  \item confirm BigQuery billing and permissions
  \item confirm dbt profile connectivity
  \item inspect \texttt{dbt\_project/target/} and \texttt{dbt\_project/logs/}
\end{itemize}
